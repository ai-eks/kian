# 微信渠道集成开发任务

## 目标

将微信作为 **Kian 的一个独立聊天渠道（chat channel）** 集成到仓库 `~/Projects/vivid` 中，并尽量复用已经抽离出的独立 NPM 包：

- 参考实现包：`/Users/lei/KianWorkspace/Kian 开发/libs/weixin-adapter`
- 包名：`@kian/weixin-adapter`

目标不是做广播 webhook，而是像 Telegram / Discord / 飞书那样，作为一个**独立聊天渠道**接入 Kian 主链路。

## 核心要求

### 1. 微信作为独立聊天渠道
需要把微信集成为与 Telegram / Discord / Feishu 同级的独立渠道，而不是复用 broadcast channel。

请完成至少这些方面的接入：

- 设置模型 / 类型定义
- 持久化设置
- IPC 接口
- 主进程运行时接线
- 渲染层设置页 UI
- 渠道初始化与生命周期
- 基本状态展示

### 2. 优先复用 `@kian/weixin-adapter`
优先采用以下方式之一：

1. 将其作为本地 workspace/package 引入到 `vivid` 仓库
2. 或将其源码以合理方式 vendoring 到 `vivid` 中

要求：

- 不要把 `openclaw/plugin-sdk` 引回 `vivid`
- 不要直接依赖 OpenClaw runtime
- 保持微信协议层与 Kian 宿主逻辑分层清晰

### 3. 先做 MVP
本次以 MVP 为主，优先支持：

- 扫码登录
- 多账号/至少单账号配置落地
- 长轮询收消息
- 文本消息发送
- 将微信消息送入主 Agent 对话链路
- 能从 Kian 回复文本回微信

如果媒体、语音、文件发送暂时无法完整接入，可以保留接口或标注 TODO，但必须说明清楚。

## 建议实施步骤

### Phase A：梳理现有聊天渠道架构
请先梳理 Telegram / Discord / Feishu 当前接入模式，定位：

- 设置结构
- 运行时初始化入口
- 入站消息 -> 主 Agent 的桥接点
- 回复输出 -> 渠道发送的桥接点

### Phase B：接入 weixin-adapter
把 `@kian/weixin-adapter` 以本地依赖方式接进来，至少保证：

- 能被 TypeScript 正常引用
- 能通过 build / typecheck
- 可以在主进程中创建 client

### Phase C：补设置与 IPC
至少补齐：

- settingsService 的微信配置读写
- shared types / validators
- preload API / renderer API
- Settings 页面里的微信配置项

建议配置项至少包括：

- `enabled`
- `stateDir`（可选）
- `accountId`（如需要）
- 登录状态信息
- 登录触发动作（开始扫码 / 轮询等待）

### Phase D：主进程聊天渠道接线
实现一个新的微信聊天渠道服务，建议命名类似：

- `electron/main/services/chatChannel/weixinTransport.ts`
- 或 `weixinChannelService.ts`

至少要能：

- 初始化 client
- 监听消息事件
- 把消息转给主 Agent
- 把主 Agent 回复回发到微信
- 管理轮询状态
- 提供错误日志

### Phase E：最小联调与文档
补齐：

- README 或开发说明
- 微信渠道接入说明
- 已完成 / 未完成项

如果条件允许，请增加至少一个：

- 最小单元测试
- 或最小集成测试
- 或至少能通过 typecheck/build
